O Node.Js

JavaScript

  • O Node.Js utiliza o JavaScript Enginie V8 (Oliveira, 2012)
  • Metódos são implementados com ou sem chamadas de retorno, do inglês callback. (HUGHES-CROUCHER; WILSON, 2012)
  • O modelo assíncrono gasta menos tempo de execução que um programa síncrono em Node.Js. (PEREIRA, 2013)
  • Evite os callbacks hell. O encadeamento de chamadas de retorno dificulta a manutenabilidade e legibilidade do código, podendo ocorrer perda de controle do que esta sendo executado e acesso à variáveis devido às trocas de escopos. (PEREIRA, 2013)

O Node.Js

Abordagem Orientação a eventos

  • Eventos são indicações de que ocorreu algo, havendo dois atores para este modelo (OLIVEIRA, 2012)
  • Produtor do evento e o consumidor do evento, do inglês. O produtor do evento não espera pela ação a ser executada pelo servidor. (JUNIOR, 2012)

O Node.Js

Thread única

  • Em Node.Js as conexões são recebidas em uma única thread invocada pelo nó de servidor de processos, do inglês node server process. (ABERNETHY, 2011)
  • Sistemas tradicionais alocam a memória RAM para cada thread criada. (ABERNETHY, 2011)
  • No apache cada conexão abre um processo filho. Se os usuários do sistema web em PHP for maior que capacidade de criar os processos filhos do Apache, o cliente entrará numa fila (POWERS, 2012)
  • A thread única pode ser facilmente escalável (POWERS, 2012)
  • Com a thread única é possível suportar um alto número de conexões, em conjunto com a repetição implícita existente no Node.Js. (TILKOV, 2010)

O Node.Js

Ciclo de eventos

  • O ciclo de eventos é uma repetição infinita que a cada interação verifica em sua fila de eventos se um determinado evento foi emitido ou se existem novos eventos (PEREIRA, 2013)
  • Utiliza o módulo EventEmitter. (PEREIRA, 2013)
  • Os bloqueios ou impasses não são características da plataforma ao realizar processamento de entradas e saídas devido a orientação a eventos. (ABERNETHY, 2011)
  • O único gargalo de um servidor Node.Js passa-se a ser a capacidade de tráfego da aplicação.(OLIVEIRA, 2012)

Simples serviço de teste de carga baseado em núvem que permite realizar um teste de estresse em seus aplicativos web ou API's com milhares de conexões simultâneas. (Sendgrid, 2014)

Conclusão

Com o desenvolvimento deste trabalho, ficou claro que a tecnologia Node.Js e suas características deve ser utilizada para solucionar o problema de milhares de conexões em servidores web pois consegue cumprir o seu objetivo a qual foi proposto com eficiência.

Um dos pontos a serem considerados, é o desenvolvimento da aplicação pois com o framework Django tem-se um rápido desenvolvimento do software capaz de fornecer uma API robusta.

No framework Express.Js o desenvolvimento é custoso para desenvolvedores menos experientes pois trata-se de uma nova abordagem e paradigma.

Conclusão

O outro ponto a ser considerado é a maturidade do framework Django, ano de 2005, e da linguagem Python.

O Node.js nasceu em 2009 e possui uma ampla comunidade open source que contribui com código e bibliotecas complementares ao sistema.

Uma melhoria em nível do protótipo Django seria a utilização de cache de consultas e dos resultados em JSON que poderiam reduzir o tempo de resposta e aumentar o número de sucessos. Este cache pode ser colocado no framework e utilizar também softwares memcached e/ou varnish em um outro servidor.(JALES; PANTOJA, 2013)
